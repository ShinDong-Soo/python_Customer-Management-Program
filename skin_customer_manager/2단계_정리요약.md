# 피부관리샵 고객 관리 프로그램 — 2단계 정리 요약 (완료)

## 1. 2단계 목표

1단계 **고객 CRUD + CSV 저장**을 확장하여, **고객-방문 기록 관계 구조**를 만들고 **입력 검증·데이터 무결성**을 강화했습니다.

```text
고객 기본 정보  →  customers.csv
방문 이력       →  visits.csv
```

---

## 5. 핵심 기능

| 기능 | 설명 |
|------|------|
| 방문 기록 등록 | 고객 ID 기준으로 visits.csv에 저장 |
| 고객별 방문 기록 조회 | customer_id로 필터링 후 목록 출력 |
| 입력 검증 | 날짜·가격·고객 ID 검증 |
| 삭제 정책 | 방문 기록 있는 고객 삭제 차단 |

---

## 6. 입력 검증 (데이터 안정성)

| 검증 | 함수 | 적용 위치 |
|------|------|-----------|
| 고객 ID 존재 확인 | `require_customer()` | 수정, 삭제, 방문 등록/조회 |
| 방문 날짜 YYYY-MM-DD | `validate_date()` | 방문 기록 등록 |
| 가격 0 이상 숫자 | `validate_price()` | 방문 기록 등록 |
| 방문 기록 존재 여부 | `has_customer_visits()` | 고객 삭제 |

### 검증 실패 메시지

```text
고객 ID를 입력해주세요.
해당 ID의 고객을 찾을 수 없습니다.
방문 날짜는 YYYY-MM-DD 형식으로 입력해주세요. (예: 2025-03-12)
가격은 0 이상의 숫자로 입력해주세요.
시술 내용은 필수 입력입니다.
해당 고객은 방문 기록이 있어 삭제할 수 없습니다.
방문 기록을 먼저 삭제한 후 다시 시도해주세요.
```

---

## 7. 고객 삭제 정책

방문 기록이 있는 고객은 **삭제 불가** (이력 데이터 보호).

```text
해당 고객은 방문 기록이 있어 삭제할 수 없습니다.
방문 기록을 먼저 삭제한 후 다시 시도해주세요.
```

방문 기록이 없는 고객만 Y/N 확인 후 삭제 가능.

## 11. 블로그 정리 포인트

> 2단계에서는 단순히 방문 기록을 추가하는 것에서 끝내지 않고,  
> CSV 기반 CRUD에서 **고객-방문 기록 관계 구조**로 확장하고,  
> 잘못된 날짜·가격·고객 ID 입력을 막아 **데이터가 깨지지 않도록 검증 로직**을 추가했다.
