# 1단계 문제 · 개선 정리

## 1. 개요

1단계 목표는 **콘솔 기반 고객 CRUD + CSV 저장**입니다.

이 문서는 1단계 개발·테스트 과정에서 발견된 **문제**와 그에 대한 **개선**을 정리한 것입니다.

---

## 2. 실행 경로 오류

### 문제

저장소 루트에서 `python main.py` 실행 시 오류 발생.

```text
can't open file '...\python_Customer-Management-Program\main.py'
[Errno 2] No such file or directory
```

- `main.py`는 `skin_customer_manager/` 폴더 안에 있음
- 현재 작업 디렉터리와 파일 위치가 달랐음

### 개선 · 대응

```bash
cd skin_customer_manager
python main.py
```

또는 루트에서:

```bash
python skin_customer_manager\main.py
```

### 교훈

> Python 실행 시 **파일 경로 = 현재 작업 디렉터리 기준**이다.

## 3. 입력값 검증 부재 (초기 1단계)

### 문제

최초 1단계 코드에는 아래 검증이 없었음.

| 항목 | 문제 |
|------|------|
| 이름 | 공백만 입력해도 등록됨 |
| 연락처 | 공백만 입력해도 등록됨 |
| 삭제 | 확인 없이 바로 삭제됨 |
| 연락처 중복 | 같은 번호 중복 등록 가능 |
| ID 입력 | 수정/삭제 시 없는 ID 처리만 메시지 출력 |

### 개선 (1단계 마무리 시 반영)

| 검증 | 내용 |
|------|------|
| 이름 필수 | `.strip()` 후 공백이면 등록 거부 |
| 연락처 필수 | `.strip()` 후 공백이면 등록 거부 |
| 삭제 확인 | `Y/N` 입력 후에만 삭제 진행 |

### 미구현 (2단계 이후 후보)

- 연락처 중복 등록 방지
- 생년월일 형식 검증
- 연락처 형식 검증 (예: 010-0000-0000)
- 수정 시 이름/연락처 필수값 재검증

## 4. 출력 코드 중복

### 문제

- `show_customers()`와 `search_customer()`에서 고객 정보 출력 코드가 **거의 동일하게 반복**
- 항목 추가·변경 시 두 곳을 모두 수정해야 함
- 검색 결과에 `생년월일`이 목록 조회보다 적게 출력되는 **일관성 문제** 존재

### 개선

- `print_customer()` 공통 함수 추가
- 목록 조회, 검색 모두 동일 함수로 출력 통일
- 목록 조회 시 **총 고객 수** 표시 추가

```python
print(f"\n총 고객 수 : {len(customers)}명\n")
for customer in customers:
    print_customer(customer)
```

## 5. 코드 실수 (push 전 수정)

### 문제

1단계 개선 작업 중 아래 실수가 발생.

| 문제 | 내용 |
|------|------|
| 중복 입력 | `add_customer()`에서 이름/연락처를 **두 번** 입력받음 |
| 들여쓰기 오류 | `load_customers()`의 `customers.append(row)` 들여쓰기 깨짐 |

### 개선

- push 전 중복 입력 코드 제거
- 들여쓰기 정상화
- 검증용 `.strip()` 입력만 유지

## 6. customers.csv 관리

### 문제

| 문제 | 내용 |
|------|------|
| 파일 중복 생성 | `customers.csv`를 repo에 두면서 `initialize_csv()`도 자동 생성 |
| 테스트 데이터 유출 | 수동 테스트 고객 정보가 `customers.csv`에 남음 |
| push 시 개인정보 | 테스트 데이터가 git에 올라갈 위험 |

### 개선

- `.gitignore`에 `skin_customer_manager/customers.csv` 추가
- 원격에는 `main.py`만 push, **로컬 데이터는 분리**
- `initialize_csv()`로 파일 없을 때 자동 생성 → **수동 파일 없어도 동작**

### 교훈

> 데이터 파일(CSV)과 코드(main.py)는 **분리 관리**하는 것이 맞다.

## 7. Git / Push 환경 이슈

### 문제

| 이슈 | 내용 |
|------|------|
| Shell 응답 없음 | Windows 샌드박스 환경에서 git 명령 실패 |
| 프로그램 실행 중 | `python main.py` 입력 대기 상태에서 다른 명령 불가 |
| 로컬 브랜치 | `branch` / `main` / `origin/main` 동기화 혼선 |

### 개선 · 대응

- Shell 권한 조정 후 push 성공
- push 전 테스트 데이터(`customers.csv`) 제외
- 프로그램 종료(`0` 또는 `Ctrl+C`) 후 git 작업

## 8. 불필요한 기능 검토

### 문제 (질문)

1단계 코드에 불필요한 기능이 있는지 검토 요청.

### 결론

- **제거할 불필요한 기능 없음** — 1단계 범위(CRUD + CSV + 메뉴)에 맞음
- 다만 아래는 **개선 여지**로 분류

| 항목 | 내용 |
|------|------|
| `customers.csv` 사전 생성 | 없어도 `initialize_csv()`가 생성 → 선택 사항 |
| 검색 출력 항목 | `print_customer()` 통합으로 해결 |
| `load_customers()` | `return list(reader)`로 줄일 수 있으나, 가독성 위해 유지 |

## 10. 1단계에서 얻은 포인트 (블로그용)

> 콘솔 메뉴와 함수 분리로 **고객 CRUD**를 구현하고,  
> CSV 파일로 **데이터를 영속 저장**했다.  
> 이후 입력 검증·출력 공통화·삭제 확인을 추가해 **사용성과 안정성**을 보강했다.

핵심 키워드:

```text
- 콘솔 CRUD
- CSV 파일 저장
- 함수 분리
- 입력값 검증 (이름/연락처)
- 삭제 확인
- 코드/데이터 분리 (gitignore)
```
