4단계 **Tkinter GUI 전환** 완료 후, 실사용·포트폴리오 관점에서 보완한 **4.5 단계** 내용을 정리한 문서입니다.

## 2. GUI 자동 테스트 부족

### 문제

- 자동 테스트는 `test_stage3.py`(DB)만 존재
- GUI는 수동 테스트에 의존
- 버튼 로직이 `gui.py`에만 있으면 회귀 테스트가 어려움

### 개선

- `services.py`에 비즈니스 로직 분리
- `test_stage4.py`로 GUI 없이 검증 가능한 항목 테스트

```text
services.py  →  prepare_customer_data(), prepare_visit_data()
             →  check_customer_deletable(), check_customer_updatable()
test_stage4.py → 등록/중복/날짜·가격/삭제 차단/미선택 테스트
```

### 테스트 항목

| 항목 | 상태 |
|------|------|
| 고객 등록 성공 | ✅ |
| 이름·연락처 필수 | ✅ |
| 연락처 중복 등록 차단 | ✅ |
| 본인 연락처 수정 허용 | ✅ |
| 잘못된 날짜 입력 | ✅ |
| 음수 가격 입력 | ✅ |
| 방문 있는 고객 삭제 차단 | ✅ |
| 미선택 수정 차단 | ✅ |
| DB 백업/복원 | ✅ |

## 3. 연락처 중복 정책 부재

### 문제

- 동일 연락처로 고객이 여러 번 등록될 수 있었음
- 실매장에서 중복 고객 관리 혼선 발생 가능

### 개선

- `get_customer_by_phone()` 추가 (`database.py`)
- `prepare_customer_data()`에서 중복 검사
- 중복 시 기존 고객 이름·ID를 안내 메시지에 표시

```text
이미 등록된 연락처입니다.
기존 고객: 홍길동 (ID: 2)
```

### 정책

- **기본: 중복 등록 금지** (DB `UNIQUE` 대신 애플리케이션 검사)
- 가족 공유 연락처 등 예외는 향후 정책으로 확장 가능
- 수정 시에는 **본인 ID 제외**하고 중복 검사

## 4. DB 백업·복구 부재

### 문제

- PC 고장·파일 손상 시 데이터 복구 방법 없음
- 실매장 운영에서 가장 중요한 공백

### 개선

- `backup.py` 추가
- GUI 상단 **DB 백업 / 백업 복원** 버튼

```text
backup/
 └─ skin_shop_2026-07-12_215500.db
```

| 기능 | 설명 |
|------|------|
| DB 백업 | 현재 `skin_shop.db`를 타임스탬프 파일로 복사 |
| 백업 복원 | 백업 목록에서 선택 후 현재 DB 덮어쓰기 |
| 복원 후 | 고객·방문 목록 자동 새로고침 |

## 5. 고객 완전 삭제 vs 비활성화

### 문제

- 방문 기록 있는 고객은 삭제 차단 → 데이터 보존에는 좋음
- 탈퇴 고객 처리 시 완전 삭제만으로는 한계

### 현재 정책 (4.5)

- 방문 없는 고객: 완전 삭제 (확인 후)
- 방문 있는 고객: 삭제 차단

### 향후 개선 (5단계 후보)

```text
customers.is_active
customers.deleted_at
→ 비활성화 처리로 전환 검토
```

## 9. 한계점 (포트폴리오용)

