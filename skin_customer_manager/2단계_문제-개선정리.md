# 2단계 문제 · 개선 정리

## 1. 개요

2단계는 **방문 기록 구조 추가**를 목표로 진행했고, 이후 **데이터 안정성·완성도 보강**까지 마무리했습니다.

이 문서는 개발·테스트 과정에서 발견된 **문제**와 그에 대한 **개선**을 정리한 것입니다.

---

## 2. 구조 설계 관련

### 문제

- 1단계에서는 `customers.csv` 하나에 모든 정보를 저장
- 방문 이력이 늘어나면 고객 정보와 섞여 관리가 어려움

### 개선

```text
customers.csv  →  고객 기본 정보
visits.csv     →  방문 이력 (customer_id로 연결)
```

- 고객과 방문 기록을 **목적별 CSV로 분리**
- `customer_id`로 관계형 구조 구현
- 고객 필드에 `created_at`(등록일) 추가

---

## 3. 테스트 데이터 혼선

### 문제

- 방문 기록을 1번만 등록했는데 조회 시 **3건**이 출력됨
- 원인: `visits.csv`에 **이전 테스트 데이터**가 남아 있었음
  - 자동 테스트 실행 데이터
  - 이전 수동 테스트 데이터

### 개선 · 대응

- `visits.csv` 헤더만 남기고 **초기화**하는 방법 정리
- `.gitignore`에 `customers.csv`, `visits.csv` 추가 → 로컬 테스트 데이터가 git에 올라가지 않도록 처리
- 테스트 전 CSV 초기화 후 진행하도록 **테스트 흐름** 정리

### 교훈

> 메뉴 7(조회)은 **해당 고객의 전체 방문 기록**을 보여주므로, 예전 데이터가 남아 있으면 함께 출력된다.

---

## 4. 고객 삭제 오해

### 문제

- 방문 기록이 없는 것처럼 보이는데 고객 삭제가 안 된다고 느낌
- 실제로는 메뉴 7 조회에서 **방문 1건**이 확인된 상태였음

### 개선 · 대응

- **삭제 정책 명확화**: 방문 기록이 있으면 고객 삭제 차단
- 안내 메시지 정리:

```text
해당 고객은 방문 기록이 있어 삭제할 수 없습니다.
방문 기록을 먼저 삭제한 후 다시 시도해주세요.
```

### 교훈

> 삭제 차단은 버그가 아니라 **의도된 데이터 보호 정책**이다.  
> 현재는 방문 기록 삭제 메뉴가 없어 CSV 직접 수정 또는 초기화가 필요하다.

---

## 5. 입력 검증 부재 (초기 2단계)

### 문제

- 방문 날짜를 `20250312`, `20250812`처럼 입력해도 저장됨
- 가격에 문자 입력 가능
- 존재하지 않는 고객 ID로 방문 등록 시도 가능
- 데이터 형식이 CSV에 그대로 저장되어 **데이터 무결성** 깨짐

### 개선

| 검증 | 함수 | 내용 |
|------|------|------|
| 고객 ID | `require_customer()` | 빈 값 / 미존재 ID 차단 |
| 방문 날짜 | `validate_date()` | `YYYY-MM-DD` 형식만 허용 |
| 가격 | `validate_price()` | 0 이상 숫자만 허용 |
| 시술 내용 | `add_visit()` | 공백 입력 차단 |

적용 위치:

- `add_visit()` — 날짜, 가격, 고객 ID, 시술 내용
- `update_customer()` — 고객 ID
- `delete_customer()` — 고객 ID
- `show_customer_visits()` — 고객 ID

## 6. 코드 품질 (1단계 → 2단계 과도기)

### 문제

- `add_customer()`에서 이름/연락처 **중복 입력** 코드 존재 (실수)
- `load_customers()` **들여쓰기 오류**

### 개선

- push 전 중복 입력 제거, 들여쓰기 수정
- 출력 공통화: `print_customer()`, `print_visit()` 함수 분리

---

## 7. 고객 삭제 정책

### 문제

- 방문 기록이 있는 고객을 삭제하면 `visits.csv`에 **고아 데이터**(orphan) 발생
- 실무에서도 이력 데이터는 함부로 삭제하지 않음

### 개선

- 방문 기록이 있는 고객 → **삭제 차단**
- 방문 기록이 없는 고객 → Y/N 확인 후 삭제 허용
- `has_customer_visits()` 함수로 방문 존재 여부 확인

### 미구현 (3단계 후보)

- 방문 기록 삭제 메뉴
- 방문 기록 수정 메뉴
- 고객 삭제 시 방문 기록 연쇄 삭제 (현재는 채택하지 않음)

## 8. 실행 환경 이슈

### 문제

| 이슈 | 내용 |
|------|------|
| 실행 경로 | 저장소 루트에서 `python main.py` 실행 시 파일 없음 오류 |
| 터미널 한글 | piped 테스트 시 한글 깨짐 (콘솔 인코딩) |
| 프로그램 대기 | `python main.py` 실행 중 터미널이 입력 대기 상태로 git 명령 불가 |

### 개선 · 대응

- 실행 경로: `cd skin_customer_manager` 후 실행
- 한글 깨짐: 코드 문제 아님, Windows 콘솔 인코딩 이슈
- 테스트 후 `0`(종료) 입력 또는 `Ctrl+C`로 종료

## 9. 2단계 전후 비교

| 구분 | 2단계 초기 | 2단계 완료 |
|------|-----------|-----------|
| CSV 파일 | customers + visits 분리 | 동일 |
| 방문 등록/조회 | 기본 CRUD | 동일 |
| 날짜 검증 | 없음 | YYYY-MM-DD |
| 가격 검증 | 없음 | 0 이상 숫자 |
| 고객 ID 검증 | 부분적 | 공통 함수화 |
| 삭제 정책 | 무조건 삭제 가능 | 방문 있으면 차단 |
| 테스트 데이터 | 혼선 가능 | 초기화 방법 정리 |

---

## 10. 2단계에서 얻은 포인트 (블로그용)

> 단순히 방문 기록 메뉴를 추가하는 것에서 끝내지 않고,  
> CSV를 목적별로 분리하고, 잘못된 입력을 막아  
> **데이터가 깨지지 않도록 검증한 프로젝트**로 발전시켰다.

핵심 키워드:

```text
- CSV 데이터 분리
- customer_id 관계 구조
- 입력값 검증
- 데이터 무결성
- 삭제 정책 (이력 보호)
```
